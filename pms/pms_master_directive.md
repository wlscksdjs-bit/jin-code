🚀 Enterprise PMS (Project Management 시stem) 통합 개발 및 고도화 지침서



1\. 프로젝트 개요 및 비즈니스 목표



본 프로젝트는 기존 엑셀(PJT 관리.xlsx 및 실행예산서) 기반의 분절되고 수동적인 관리 체계를 완전히 탈피하여, 수주 기획 -> 원가 분석 -> 실행 예산 통제 -> 전자 결재 -> 인력/일정 관리 -> 경영진 대시보드로 이어지는 전사적 프로젝트 관리 웹 애플리케이션(Enterprise PMS)을 구축하는 것을 궁극적인 목표로 한다.



1.1 기존 엑셀 관리의 한계점 (Pain Points)



데이터 사일로 현상: 각 담당자의 PC에 파일이 산재되어 있어, 최신 버전 확인이 어렵고 데이터 취합에 막대한 시간이 소요됨.



휴먼 에러 리스크: 복잡한 수식이 연결된 엑셀 특성상, 실수로 수식이 지워지거나 참조가 어긋나 원가 계산에 치명적인 오류가 발생할 위험이 상존함.



실시간 통제 불가: 예산 초과나 일정 지연이 발생해도, 엑셀 취합 주기가 도래하기 전까지는 경영진이 선제적으로 인지하고 대응하기 어려움.



1.2 시스템 도입 기대 효과 (Expected Benefits)



단순한 데이터 입력을 넘어 엑셀의 복잡한 수식과 로직을 시스템 내에 완전히 내재화한다. 엄격한 데이터베이스 트랜잭션을 통한 버전 관리, 이상 감지 시스템을 통한 선제적 리스크 대응(Alert), 모바일 접근성 강화를 통한 현장 업무 효율성 극대화, 그리고 완벽한 엑셀 입출력 기능을 통해 기존 사용자들의 거부감을 최소화하면서도 고도화된 데이터 경영 환경을 제공한다. 특히 기존에 수기로 관리하던 수많은 엑셀(제안서용 견적, 실행예산서 등)을 시스템에 원클릭으로 업로드 및 자동 등록하는 기능을 통해 과거 데이터의 마이그레이션과 실무 도입의 허들을 낮춘다.



2\. 시스템 아키텍처 및 기술 스택 상세 (권장)



안정적인 금융/원가 데이터 처리와 향후 시스템 확장을 고려하여 아래와 같은 기술 스택과 아키텍처 설계 원칙을 적용한다.



Backend: Python 3.11+ 기반 (Django + Django REST Framework)



선정 사유: 엑셀 파일 파싱 및 대규모 데이터 전처리(pandas, openpyxl)에 최적화되어 있으며, Django ORM을 활용하여 복잡한 재무 데이터의 트랜잭션을 안전하게 제어할 수 있음. 첨부된 다단 헤더 형태의 복잡한 엑셀(실행예산서 등)을 처리하기 위한 강력한 데이터프레임 조작 기능을 제공함.



아키텍처: 서비스 레이어 패턴(Service Layer Pattern)을 적용하여, API View와 비즈니스 로직(원가 계산 등)을 엄격히 분리하여 테스트 용이성을 확보함.



Frontend: React.js (또는 Vue.js 3) + TypeScript



선정 사유: 상태 관리와 컴포넌트 재사용성이 뛰어나며, 복잡한 데이터 그리드 및 간트차트 구현에 유리함. TypeScript를 도입하여 원가 관련 숫자 데이터의 타입 안정성을 컴파일 단계에서 보장.



UI/UX: Tailwind CSS를 적극 활용하여 PC, 태블릿, 스마트폰 등 다양한 디바이스 환경에 유연하게 대응하는 반응형(Responsive) 웹 디자인 구현.



Database: PostgreSQL 15+



선정 사유: 고도화된 동시성 제어(MVCC)를 지원하여 다수의 사용자가 동시에 원가/결재 데이터를 수정하더라도 데이터 무결성(ACID)을 강력하게 보장. JSONB 필드를 활용하여 비정형화된 엑셀 메타데이터도 유연하게 수용 가능.



인프라 및 기타 도구:



비동기 처리: Celery \& Redis (대용량 엑셀 업로드/다운로드 백그라운드 처리, 야간 스케줄링을 통한 지연 알림 메일 일괄 발송).



보안/인증: JWT(JSON Web Token) 기반의 Stateless 인증 체계 및 HTTPS/TLS 1.2 이상 적용.



3\. 핵심 모듈별 상세 구현 지침 (Excel 수식 연동 및 비즈니스 로직)



Module A: 수주 기획 및 파이프라인 관리 (Pre-Sales \& Bidding)



$$참조 엑셀: 대기, 수처리, BHI, 도레이, KTC 시트, 제안서용 견적$$



기능 요구사항: 영업 파이프라인 상의 예상 프로젝트 목록을 등록하고, 목표 수주 금액 대비 상세 원가 분석을 수행하여 최적의 입찰 전략을 도출한다.



첨부 엑셀(견적/보고서) 자동 등록 연동: 기존에 관리되던 '제안서용 견적' 또는 '대기/수처리 프로젝트별 실행예산 보고' 등의 엑셀 파일을 시스템에 업로드하면, 해당 파일 내의 프로젝트명, 수주금액, 직접원가(기계, 전/계장 등), 이익률 등을 시스템이 자동 파싱하여 PJT 목록 및 세부 파이프라인으로 일괄 등록시킨다.



상태 관리 (Pipeline Stages): 타겟 발굴 -> 제안 준비 -> 입찰(Bidding) -> 수주 성공(Won) / 실파(Lost) 단계로 상태(Status) 값을 명확히 분리하여 관리.



수식 및 로직 구현 (필수 적용):



직접원가 계 = 기계 + 전/계장 + 촉매 + 구매품 + 기타(현장 경비, 보험료 등)



간접비 (A) = 직접원가 대비 특정 비율(%) 적용 (사내 정책에 따른 변동 변수화 필요)



판관비 (B) = 매출(예상 수주액) 대비 특정 비율(%) 적용



영업이익 (C) = 수주금액 - (직접원가 계 + 간접비 + 판관비)



이익률(%) = 영업이익 / 수주금액



고도화된 버전 관리 (Version Control \& Snapshot):



영업 과정 중 입찰가, 원가 항목, 네고(Negotiation) 조건 등이 수정될 때마다 기존 데이터를 덮어쓰는 것(Overwrite)을 엄격히 금지함.



수정 시 반드시 v1.0, v1.1, v2.0 형태의 스냅샷 레코드를 DB에 새롭게 생성(Insert)함.



UI 측면에서는 '버전 비교(Compare) 모드'를 제공하여, 과거 버전(예: 최초 제출가)과 현재 버전(예: 최종 네고가) 간의 원가 증감액 및 이익률 변화를 한눈에 볼 수 있는 차이(Diff) 테이블을 제공해야 함.



Module B: 실행원가 및 예산 통제 시스템 (Project Cost \& Budget Tracking)



$$참조 엑셀: 실행예산서 (2) 시트$$



기능 요구사항: 수주가 확정된 PJT를 대상으로 항목별(기계, 배관, 전기, 토목 등), 협력업체별 세부 실행예산을 편성하고, 실시간으로 비용의 흐름(발주-기성-지급)을 엄격하게 통제한다.



실행예산서 엑셀 다이렉트 업로드: '실행예산서' 형태의 엑셀 포맷을 그대로 업로드하면, 다단으로 구성된 헤더(항목, 업체명, 계정명, 최초실행원가, 네고반영가 등)를 매핑 엔진이 자동 인식하여 예산 계획을 DB에 항목별로 자동 편성한다. 특히 1월\~12월 등 '비용 사용계획' 열을 시계열 데이터로 정규화하여 저장한다.



데이터 구조 및 필드 정의: 항목(WBS 매핑), 발주처(업체명), 회계 계정명, 최초실행원가(A), 네고반영가(C), 증감(C-A).



비용 트래킹 및 통제 로직 (핵심 엔진):



계획 예산 = 최종 승인된 네고반영가(C) (이는 결재를 거치지 않고는 임의 수정 불가함).



기사용금액 = 과거 결재가 최종 완료되어 협력업체에 지급이 승인된 누적 총액.



금번사용금액 = 현재 월(또는 특정 회차)에 기성 청구되어 결재가 진행 중인(Pending) 임시 금액.



사용예정금액(잔여) = 향후 투입되어야 할 미발주/미청구 잔액. (필요 시 현장 PM의 판단에 따라 잔여 예상액을 수동 조정할 수 있는 'ETC(Estimate To Complete)' 기능 추가 고려).



예상 최종 금액 (EAC) = 기사용 + 금번사용 + 사용예정금액.



원가 절감/초과 (Variance) = 계획 예산 - 예상 최종 금액.



시각적 통제 UI: 계산된 분산(Variance)이 음수(예산 초과)일 경우 화면상에 강렬한 적색(Red) 하이라이트 및 경고 아이콘 표시. 양수(예산 절감)일 경우 청색(Blue)으로 표기하여 직관적 인지 보장.



Module C: 전자 결재 워크플로우 (Digital Approval System)



기능 요구사항: 예산의 집행, 변경, 기성 청구 등 재무적 변동이 일어나는 모든 행위에 대해 시스템 내 투명한 결재 이력을 남긴다.



결재 유형 (Forms): 1) 당월 현장/본사 비용 내부 품의, 2) 외주 협력사 기성 지급 품의 (세금계산서 연동), 3) 고객사 대금 청구 품의.



상태 변이 및 트랜잭션 로직 (원자성 보장 필수):



결재 프로세스: 임시저장 -> 결재 상신 -> 검토중(중간 결재자) -> 최종 승인완료 / 반려.



핵심 트리거 (Data Trigger): '협력사 기성 지급 품의' 문서가 



$$최종 승인완료$$



&#x20;상태로 변경되는 즉시, Module B의 데이터 테이블에 접근하여 해당 항목의 금번사용금액 데이터를 0으로 초기화하고, 그 금액만큼 기사용금액에 합산(Update)해야 함.



예외 처리: 만약 DB 업데이트 과정에서 네트워크 오류나 데드락(Deadlock)이 발생할 경우, 비용 데이터가 증발하거나 중복 합산되지 않도록 완벽한 롤백(Rollback) 처리가 구현되어야 함.



위임 및 대결 기능: 주요 결재자(경영진, 본부장 등) 부재 시 사전에 설정된 대리자에게 결재 권한이 위임되는 기능 포함.



Module D: 인력 배치 및 리소스 통제 (Resource Allocation \& Capacity Planning)



기능 요구사항: PM, 설계, 구매, 공사 등 부서별 가용 인력을 파악하고, PJT별로 적정 투입 기간과 M/M(Man/Month)를 배분하여 인력 운용의 효율성을 극대화한다.



직무 역량 기반 매핑: 단순한 인원수 맞추기를 넘어, 각 직원의 직무(Role)와 숙련도(Grade) 데이터를 기반으로 특정 Task에 적합한 인력을 필터링하여 배정할 수 있도록 지원.



고급 이상 감지 로직 (Conflict Detection):



중복 투입 (Over-allocation): 특정 직원이 동일한 기간(주차/월 단위)에 2개 이상의 PJT에 중복 투입되어 총 합산 투입률이 100%(예: A프로젝트 70%, B프로젝트 50% = 120%)를 초과할 경우, 시스템 상에 경고(Warning) 플래그를 발생시킴.



필수 인력 누락: PJT 진행 단계상 반드시 필요한 인력(예: 시공 단계가 시작되었으나 현장소장 및 안전관리자가 미배치된 상태)일 경우, 대시보드 및 해당 PM에게 치명적 알림(Alert) 발생.



Module E: 간트차트 및 진도 관리 (Gantt Schedule \& WBS)



기능 요구사항: 주요 마일스톤(설계 승인, 자재 입고, 시공 완료, 시운전 등)에 대한 주차별 일정을 시각화하고, 계획 대비 실적 진도율을 관리한다.



UI/UX 고도화: Frappe Gantt, DHTMLX Gantt 또는 Bryntum Gantt 등의 고성능 JavaScript 라이브러리를 적용하여, 마우스 Drag \& Drop 만으로 일정을 이동시키고 막대 길이를 늘려 기간을 연장할 수 있는 직관적 조작 지원.



스케줄링 핵심 기능:



Baseline(기준선) 관리: 최초 승인된 계획 일정(Plan)을 회색 점선 등으로 고정 표시하고, 그 위에 실제 진행률(Actual %)을 컬러 바(Bar)로 겹쳐서 표현하여 지연 여부를 즉시 파악.



종속성(Dependencies): FS(Finish-to-Start), SS(Start-to-Start) 등 작업 간의 선후행 관계 설정. 선행 작업(예: 설계 승인 지연)이 늦춰질 경우, 시스템이 후행 작업(발주/시공)의 일정을 자동으로 밀어내는 스케줄 자동 연기 엔진 구현.



Module F: 경영진 대시보드 및 전사 이상 감지 센터 (Executive Dashboard \& Alerts)



기능 요구사항: 경영진 및 C-Level이 기업의 전체 프로젝트 포트폴리오 건전성을 한눈에 파악하고, 리스크에 선제 대응할 수 있는 요약 정보 및 조기 경보를 제공한다.



대시보드 위젯 구성:



기업 전체 올해 수주 목표 대비 현재 달성률 (게이지 차트).



주요 프로젝트별 수주 이익률 vs 현재 예상 이익률 변동 추이 (라인 차트).



부서별/개인별 리소스 부하 상태 및 가동률 현황 (히트맵 형태).



현재 결재 대기 중인 주요 품의 건수.



이상 감지 알림 시스템 (Proactive Alert System):



비용 리스크: 실행 예산 항목 중 원가 초과가 단 1원이라도 예상되는 즉시, 담당 PM과 경영진에게 시스템 내 우측 하단 Toast 팝업 및 긴급 이메일 자동 발송.



일정 리스크: 간트차트 상의 'Critical Path(주공정)'에 있는 마일스톤이 계획 대비 2주(14일) 이상 지연 상태로 방치될 경우 적색 경고 알림 발생.



4\. 공통 UI/UX 및 특수 요구사항 (매우 중요)



본 시스템은 수십 년간 엑셀에 익숙해진 실무자들이 거부감 없이 사용할 수 있도록 엑셀의 장점(가시성, 조작성)을 웹에서 완벽히 재현해야 한다.



엑셀과 동일한 조작감 (틀고정 및 고성능 데이터 그리드):



비용 및 예산 관리 화면은 수백 개의 행(품목)과 수십 개의 열(월별 금액, 통계치)을 포함한다.



브라우저에서 가로/세로 스크롤을 하더라도 상단 헤더(항목명)와 좌측 기준 열(예: 시스템명, 프로젝트명 등 최대 3열)이 화면에 고정되는 '틀고정(Freeze Panes)' 기능을 반드시 구현해야 한다.



대량의 DOM 렌더링 성능 저하를 막기 위해 CSS position: sticky 만으로는 한계가 있으므로, AG Grid, Handsontable 또는 Toast UI Grid와 같은 가상 스크롤(Virtual Scrolling) 기반의 전문 DataGrid 라이브러리 사용을 강력히 권장한다.



완벽한 엑셀 데이터 입출력 (Export, Import \& Parsing):



양식 다운로드: 신규 PJT 원가 분석 시, 시스템에 미리 등록된 표준 엑셀 템플릿 양식을 다운로드하는 기능.



첨부 엑셀(기존 양식) 원클릭 업로드 및 자동 등록 (Import Mapping Engine):



다중 양식 지원: 현재 실무에서 사용 중인 다양한 형태의 엑셀 파일(예: 제안서용 견적, 실행예산서, KTC/도레이/BHI 등 각 프로젝트 시트, 장표\_Item/Weight 등)을 시스템 양식으로 강제 변환할 필요 없이 그대로 업로드할 수 있는 지능형 매핑 엔진을 백엔드에 탑재한다.



스마트 파싱(pandas 활용): 파일명이나 시트명을 시스템이 자동 인식하여, 해당 파일이 '수주 기획(Module A)'을 위한 견적 데이터인지, '실행 원가(Module B)'를 위한 예산 데이터인지, 혹은 '대시보드(Module F)'를 위한 장표 데이터인지 판별하여 DB에 적합한 구조로 일괄 Insert/Update 처리한다.



동적 및 다단 헤더 인식: '최초 실행원가(2025.02) A', '네고 반영가 C'와 같이 줄바꿈이나 병합이 포함된 다단 헤더를 유연하게 파싱하며, '전년누계', '1월\~12월' 등 횡렬로 나열된 비용 사용계획 데이터를 정규화하여 DB의 시계열 테이블로 완벽히 매핑한다.



실시간 검증 및 피드백 모달: 업로드 중 데이터 타입 오류나 누락(예: 단가나 수량이 텍스트로 기입된 경우) 발견 시, 시스템이 임의로 건너뛰지 않고 직관적인 모달 창을 통해 "실행예산서 15행 '송풍기' 항목의 네고 단가 데이터 형식이 잘못되었습니다"와 같이 명확한 피드백을 제공하여, 사용자가 즉각 수정 후 재업로드할 수 있도록 지원한다.



보이는 대로 다운로드 (Export): 현재 브라우저 화면에 사용자가 필터링, 정렬해 둔 데이터 그리드 상태 그대로를 엑셀 파일(.xlsx)로 생성하여 내려받는 기능.



반응형 웹 및 현장 모바일 최적화 (Mobile First for Field Workers):



건설/플랜트 현장의 특성상, 현장 담당자가 스마트폰이나 태블릿으로 시스템에 접속하는 비율이 높다.



PC용 복잡한 그리드 화면은 모바일에서 무리하게 표출하지 않고, '카드(Card) 형태'의 레이아웃으로 변환하여 모바일 친화적 UI를 제공해야 한다 (Tailwind CSS의 md:, lg: 반응형 유틸리티 클래스 적극 활용).



현장에서 즉시 사진을 촬영하여 기성 내역에 첨부하거나, 모바일 화면에서 원터치로 결재 상신 및 승인이 가능하도록 동선을 최소화한다.



엄격한 계정별 권한 통제 (RBAC - Role Based Access Control):



최고관리자 (Admin): 시스템 마스터 데이터(코드, 사용자, 권한 그룹) 관리.



경영진 (Executive): 모든 데이터 읽기 권한(Global Read-only), 대시보드 접근, 원가 초과 승인 및 최종 결재 승인 권한.



PM (Project Manager): 본인이 배정된 PJT에 한정하여 원가 편성/수정, 일정 조정, 리소스 배분 및 관련 품의 상신 권한. (타 PJT 접근 불가).



일반 담당자 (Team Member): 배정된 세부 Task(WBS)의 진행률(%)만 업데이트 가능. 보안 지침: 하위 담당자에게는 전체 프로젝트의 총 수주액, 이익률, 협력업체별 단가 등 민감한 금액(원가) 정보는 완전히 마스킹(\*\*\*) 처리하거나 해당 메뉴 자체의 접근을 차단해야 한다.



5\. 단계별 개발 및 테스트 지시사항 (Sprint \& Action Plan)



방대한 시스템이므로 빅뱅 방식의 오픈을 지양하고, 애자일(Agile) 방식의 스프린트로 모듈을 분할하여 개발 및 테스트를 진행한다.



Phase 1: 인프라 기반 세팅 및 엑셀 데이터 자동 업로드(수주 파이프라인) 연동 (Module A \& Auth)



개발: DB 물리 모델 설계 (Users, Roles, Projects, Pipeline, Versions 테이블). JWT 기반 인증 시스템 구축. 엑셀(제안서용 견적, 대기, 수처리 시트 등) 구조를 분석하여 DB 스키마 매핑 엔진 개발 및 이익률 계산 수식을 백엔드 로직에 내재화.



Test Point: 첨부된 다양한 양식의 엑셀 파일을 업로드했을 때, 다단 헤더를 정확히 인식하고 수식이 백엔드의 계산 결과값과 일치하게 DB에 자동 등록되는지 검증. 데이터 수정 시 새 버전(v1.1)이 생성되고 구 버전(v1.0)이 보호되는지 확인.



Phase 2: 실행예산 통제 및 결재 트랜잭션 핵심 엔진 (Module B \& C)



개발: Budget(예산) 및 Expense(비용) 모델 구축. 엑셀(실행예산서 시트)의 데이터 그리드 UI를 프론트엔드에 틀고정 기능과 함께 구현 및 해당 양식 업로드 자동화 로직 구현. 상태 변이 머신 기반의 전자결재(Approval) 워크플로우 엔진 개발.



Test Point: 협력사 기성 품의 승인 이벤트를 발생시켜, 금번사용금액이 기사용금액으로 이관되고 원가 절감/초과 액수가 실시간으로 재계산되는지 확인. 고의로 예외(Exception)를 발생시켜 트랜잭션 ACID 롤백 기능이 철저히 작동하는지 검증.



Phase 3: 간트차트 및 리소스 할당 시각화 (Module D \& E)



개발: WBS 일정 모델 및 리소스(M/M) 모델 구축. 프론트엔드에 JavaScript 간트차트 컴포넌트 이식. 사용자별 일정을 한눈에 보는 리소스 히트맵 뷰(Heatmap View) 개발.



Test Point: 한 명의 직원을 동일 주차에 복수의 프로젝트에 중복 투입하여 합산치가 100%를 초과하도록 조작해보고, 프론트엔드/백엔드 양쪽에서 완벽히 에러 메시지를 반환하고 저장을 차단하는지 테스트. 의존성이 걸린 선행 Task 연기 시 후행 Task 밀림 자동화 테스트.



Phase 4: 대시보드 시각화 및 통합 알림 센터 (Module F)



개발: 경영진을 위한 Chart.js / Recharts 기반의 시각적 대시보드 위젯 구현. Celery 스케줄러를 활용한 배치(Batch) 기반의 이상 감지 로직(비용 초과, 일정 지연 감시) 구동. 웹 소켓(WebSockets) 또는 SSE를 이용한 실시간 브라우저 팝업 알림 연동.



Test Point: 시스템 내 다양한 모듈(예산 초과, 인력 충돌)에서 발생한 이상 징후가 대시보드 상에 즉각 반영되는지 확인. 모바일(가장 작은 해상도 디바이스) 환경에서 결재 승인 및 그리드 조회 등 UI 요소가 깨지지 않는지 완벽 검증.



Phase 5: 통합 E2E(End-to-End) 테스트 및 사용자 안정화



개발: 시스템 전 구간 E2E 테스트. 수주 엑셀 업로드 자동 등록 -> 버전 갱신 -> 실행 원가 엑셀 편성 -> 비용 집행 및 결재 -> 대시보드 통계 반영 및 일정 종료로 이어지는 전체 프로젝트 생애 주기(Life Cycle) 시뮬레이션 검증. 실무자 대상의 UAT(사용자 인수 테스트) 및 매뉴얼 배포.
